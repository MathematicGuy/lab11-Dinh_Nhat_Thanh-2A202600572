Guardrail Decision Matrix

Quick reference for the Day-11 banking agent: which layer should catch which kind of risk, what signal it uses, and what latency price you pay.

Layer Best for catching Typical signal Latency profile Failure if overused
Input rail Direct prompt injection, off-topic requests, obvious policy violations, malformed or abusive traffic Regex, keyword/rule filters, topic allowlist, cheap classifiers Fast usually no extra LLM call False positives block legitimate users before the model can help
LLM rail Subtle safety judgments, ambiguity, quality scoring, hallucination or unsafe intent not obvious from rules alone Judge model score, structured policy rubric, confidence threshold Slower adds one or more model calls High cost and latency if used on every request
Output rail PII leakage, secret leakage, unsafe instructions, formatting constraints, banned content in the final response Pattern scan, redaction rules, moderation, LLM-as-judge on output text Fast for regex/redaction; Slower if model judging Catches leakage late, after tokens were already generated
HITL High-stakes or low-confidence actions, novel attacks, disputed decisions, sensitive transactions Confidence score, risk score, amount threshold, repeated failures, anomaly flags Slowest human queue or manual review Workflow bottlenecks if escalation thresholds are too low

Routing rule of thumb

  • Put cheap, high-precision checks first.
  • Use LLM judging only where rules are too brittle.
  • Use output rails because some failures only appear after generation.
  • Reserve HITL for high-harm or low-confidence cases, not as a bandage for weak automation.
Day-11 mapping: src/guardrails/input_guardrails.py should reject obvious attacks early; src/guardrails/output_guardrails.py should catch leakage or unsafe completions; src/hitl/hitl.py should handle the residual risky cases that automation should not auto-approve.
